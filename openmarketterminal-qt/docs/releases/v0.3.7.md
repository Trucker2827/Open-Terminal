# v0.3.7 — Financial-analysis skills in the in-app AI

The in-app assistant can now produce institutional-style analyst work product —
comps, DCF, earnings notes, LBOs, and 3-statement models — by following built-in
methodology "playbooks" and orchestrating the app's own **free** SEC data tools.
Read-only: it drafts analysis into the Report Builder; it never trades.

## New

- **Claude Code can now drive the whole terminal.** The MCP bridge used by the Claude
  Code adapter previously exposed only the alphabetically-first 50 tools (a prompt-size
  cap meant for the token-limited *in-app* assistant), so most of the terminal —
  `get_quote`, `get_observations`, the `edgar_*` filings tools, `report_*`, `notebook_*` —
  was callable by name but invisible to discovery. External clients that do their own tool
  selection now get the **full catalog (605 tools)** via an opt-in uncapped path; the
  in-app assistant's prompt budget is unchanged. The terminal is now fully scriptable from
  an external AI, not 8% of it.
- **Analysis playbooks baked into the AI.** Ask the assistant for any of these and it
  follows a proper methodology using `edgar_get_financials`, `edgar_calc_multiples`,
  `get_quote`, and the Report Builder — no paid data connectors:
  - **Comps** — comparable-company analysis: peer set, operating metrics, multiples,
    median/quartile stats, implied value.
  - **DCF** — intrinsic valuation: FCF build, WACC, terminal value, equity bridge, and
    a WACC × terminal-growth sensitivity table.
  - **Earnings / filing note** — YoY financials, MD&A and risk read, guidance, key risks.
  - **LBO** — Sources & Uses, operating model, debt schedule with cash sweep, exit, and
    sponsor returns (IRR / MOIC) with a sensitivity grid.
  - **3-statement model** — linked Income Statement / Balance Sheet / Cash Flow with the
    balance check (A = L + E) and cash tie-out enforced, plus credit metrics.
- **AI-authored notebooks.** Ask the in-app AI for an analysis you want to run yourself
  ("make me a DCF notebook for AAPL I can edit") and it writes a runnable Python notebook
  into the Notebooks screen and opens it — you run and tweak the cells (edgartools/pandas
  in the kernel). The "AI analysis workspace."
- **Company Fundamentals notebook** in the Notebooks library — pulls a company's real
  SEC financials (revenue, margins, net income, EPS, ROE + a multi-year trend) via
  edgartools; set a ticker and Run All.
- **Slash commands in chat:** `/comps`, `/dcf`, `/earnings`, `/lbo`, `/3statement`
  (each takes a ticker, e.g. `/comps AAPL`). A command without a ticker shows a usage
  hint; anything else you type is sent as a normal message.
- **Local / offline models now actually drive the tool loop.** Premium Claude remains the
  recommended engine, but weak local models (e.g. llama 3.3 via Ollama) are now a usable
  fallback instead of stalling:
  - **Lean prompt for local models.** The heavy, Claude-tuned default system prompt
    (~2,500 words with the analysis playbooks) was measured to *break* small models —
    they stop emitting structured tool calls and dump bare JSON into the chat. Local
    models (Ollama, or any OpenAI-compatible endpoint on localhost) now get a tight
    ~90-word prompt and skip the playbook appendix. Measured improvement across two local
    models: both went from failing the multi-tool task under the full prompt to completing
    it cleanly with structured calls under the lean prompt. A capable model (llama 3.3 70b)
    also completes it on the app's default Tool-RAG path (discover via `tool_list`, then
    call). Smaller models (e.g. an 8B) improve but may not manage multi-step RAG discovery
    — for those, turning **Tool-RAG off** (so all tools are pre-loaded) is the better mode.
    Capability still scales with the model; premium Claude remains the recommended engine.
  - **Bare-JSON tool-call salvage.** When a model emits a tool call as plain-text JSON
    (`{"name":…,"arguments":…}`, `{"type":"function",…}`, a `\`\`\`json` block, or even a
    list of stringified call objects) instead of a structured `tool_call`, the loop now
    recognizes and executes it instead of leaving it dead in the chat. Guarded to known
    tool names so a legitimate JSON answer is never mistaken for a tool call; premium
    Claude (structured calls) is unaffected.
  - **Dev harness:** `scripts/dev/probe_local_model_toolloop.py` drives any Ollama model
    through a faithful multi-turn tool loop (real prompt, structured tools, multi-tool
    task, optional Tool-RAG discovery) and A/Bs full vs lean prompt — used to measure the
    above and re-checkable against any locally loaded model (`--model <tag>`).
  - **Scope:** this makes plain tool-driven chat work on local models. The deeper analysis
    *playbooks* (and so `/comps`, `/dcf`, etc. at full fidelity) are still written for and
    tuned to premium Claude — on a local model those run best-effort without the bundled
    methodology. Premium Claude (Sonnet 4.x) remains the recommended engine for analysis.

## Fixed

- **Quick Chat now follows the active model.** The floating Quick Chat bubble could keep
  using the previously-loaded provider (e.g. the Ollama default) after you switched
  models. It now reloads on `llm.provider_changed` — same as the AI Chat tab — so both
  always use the model selected in Settings → LLM Config.

## Notes

- Requires an LLM provider configured in **Settings → LLM Config** (the playbooks call
  tools, so tool use must be on).
- All data comes from **SEC EDGAR** (free, public-domain) and the app's existing market
  tools — no third-party/paid connectors are added.
- Methodology is adapted from Anthropic's *Claude for Financial Services*
  (github.com/anthropics/financial-services, Apache-2.0); see
  `resources/ai_skills/NOTICE.md`. Output is draft analyst work product for review by a
  qualified professional — not investment advice.
